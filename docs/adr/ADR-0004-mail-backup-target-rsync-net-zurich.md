# ADR-0004: Mail Backup Target — rsync.net Zurich, Phase-1 Single Off-Site

## Status

**Accepted** — 2026-05-07

## Context

Following the institutional mail rollout (`docs/PLAN-mail-self-hosted.md`,
`docs/runbooks/MAIL-IMPLEMENTATION-STATUS.md`), the Mailcow stack on
`lince` (5.182.33.93, Contabo) holds two classes of state that cannot be
recovered from any other source:

- **Mail content** — `vmail/` Maildir (per-user mailbox content), MySQL
  dump (Mailcow account/alias schema), Redis (rspamd state), Postfix
  queue, SOGo data, generated `mailcow.conf`. Lives under
  `/opt/mailcow-dockerized/data/`.
- **Mail configuration** — domains, mailboxes, aliases, DKIM keys
  generated by Mailcow itself (Mailcow's DKIM keys are stored in MySQL,
  not the well-known DNS CNAMEs which are Postmark-side and survive
  independently).

A failure of `lince` (Contabo VPS loss, disk corruption, ransomware
encrypting both running stack and any local backups) has no recovery
path without an off-site backup target.

A target was therefore required for Phase-1. Five candidates were
evaluated: rsync.net (Zurich), Backblaze B2 (Amsterdam region), Storj,
Hetzner Storage Box, and a self-hosted secondary VPS. The full
comparison lives in the conversation history (2026-05-07 thread). The
salient axes were:

1. **Jurisdiction** — rbxsystems.ch positions itself as a Swiss
   institutional brand. Backup state in US-based providers (CLOUD Act
   exposure) creates a coherence mismatch.
2. **Immutability against ransomware** — backup must survive a
   compromise of `lince` itself. Both ZFS snapshots (provider-side) and
   S3 Object Lock (client-configured) can provide this; the latter has
   higher implementation risk.
3. **Operational model** — the target should fit naturally into
   SSH/Ansible/Borg/Restic toolchains already in use, not require a new
   mental model.
4. **Cost shape** — at Phase-1 volumes (1–10 GB), absolute costs are
   trivial. Cost differences become material only at >50 GB sustained.

## Decision

The Phase-1 off-site backup target for the Mailcow stack is
**rsync.net, Zurich datacenter**, on a **standard account**.

### Account-type distinction (do not get this wrong)

rsync.net offers two account types relevant here:

| Account type | Pricing | ZFS snapshots | Operational surface |
|---|---|---|---|
| **Standard** | ~$0.025/GB-mo pay-as-you-go, ~$0.012–0.015/GB-mo with annual prepay | **Included, provider-side, default 7-day retention; extension negotiable** | Full SSH access, all rsync.net features, full support |
| **Borg-only (discounted)** | ~50% of standard | **Not part of the included service** for this tier — customer-managed only | Restricted: only Borg/Restic protocol allowed; no general SSH |

RBX takes the **standard account** explicitly:

- ZFS snapshots provider-side are a primary control against ransomware
  and operator error. Losing them to optimize for storage cost is
  unacceptable at this stage.
- Full SSH operational surface keeps the toolchain coherent with the
  rest of RBX infrastructure (Ansible-over-SSH everywhere).
- Swiss-coded brand alignment: Zurich datacenter, full-service Swiss
  storage relationship.
- Borg or Restic still runs **on top** of the standard account as the
  client-side encryption/deduplication engine. The discount tier is
  **rejected** because it conflates "use Borg" with "give up snapshots".

RBX selects the standard account for Phase-1. Changing this selection
requires an ADR update.

### Engine

The encryption/deduplication engine is **Borg or Restic** (final choice
deferred to the implementation ADR, but constrained to one of these
two).

Both support client-side encrypted, deduplicated backups. The
implementation must prove an append-only or equivalent immutability
posture for the selected engine before production use.

Borg leans toward simpler semantics and better dedup; Restic has more
backends (would help if a future 3-2-1 secondary needs S3). The
selection happens during implementation based on a small spike against
a real Mailcow `data/` snapshot, and the spike must validate the
immutability posture for the chosen engine.

### Retention

The retention pyramid is layered: client-side (Borg/Restic policy) on
top of provider-side (ZFS snapshots).

**Client-side (Borg/Restic):**

```
--keep-daily   7
--keep-weekly  4
--keep-monthly 6
```

Total: ~17 distinct restorable points spanning ~6 months. Aligned with
the existing 30-day-retention precedent for PowerDNS pg_dump
(`docs/PLAN-dns-email-architecture.md`), but layered to give weekly and
monthly granularity at distance.

**Provider-side (ZFS snapshots, rsync.net):**

Default 7-day daily snapshots from the rsync.net side, retained
independently of the customer-managed Borg/Restic repository. These are
the line of defence against a compromised client that runs `borg
prune` maliciously.

Retention beyond 7 days at the ZFS layer is not requested for Phase-1.

### Restore drill — quarterly, mandatory

A restore drill is **mandatory** every quarter. The drill is not a
target choice; it is a process commitment that makes the target
choice meaningful.

The drill restores `data/` from the most recent backup into a scratch
host (or scratch directory on `lince` itself, isolated from the live
stack) and verifies:

- Mailcow stack starts against the restored data
- A small known mailbox (the test/`postmaster` account) opens and
  shows expected content
- The restored Mailcow configuration is internally consistent
  (domain, mailbox, and alias counts match the IaC inventory; no
  half-restored objects)
- Outbound authentication is verified through a real delivered test
  message and its Authentication-Results headers — not by comparing
  internal Mailcow state against any DNS values

The restore drill runbook **must distinguish Postmark-side DKIM /
SPF / DMARC / Return-Path** (the relevant authentication path for
current Phase-1 outbound, anchored in DNS CNAMEs and TXT records)
**from any Mailcow-side DKIM state in MySQL** (dormant under
Option B since outbound goes 100% through the Postmark relay, and
not asserted in DNS). Comparing Mailcow's internal DKIM keys
against Postmark's public CNAMEs would produce a false-negative
drill result; the verification point is end-to-end delivery
authentication observed at the recipient, not internal key
equality.

A drill that has not been run within 90 days invalidates the
"production-grade" status of the mail stack, regardless of how well
the live system appears to function. This is enforced by runbook, not
by tooling, in Phase-1.

### Secondary 3-2-1 target — Backblaze B2 Amsterdam, deferred

A second off-site target is **explicitly deferred**, not abandoned. The
intended secondary is **Backblaze B2 in the Amsterdam (EU) region**
with Object Lock for immutability, written via Restic over the S3
gateway.

The 3-2-1 secondary is **re-evaluated** when **any** of the following
trigger conditions hold:

1. **First external client uses an RBX-hosted email address for
   relevant data.** "Relevant" means: customer correspondence,
   contracts, or any content where loss imposes contractual or legal
   exposure.
2. **Mail volume crosses 50 GB.** At this scale, restore time from a
   single target becomes operationally non-trivial, and the cost
   delta between providers becomes material.
3. **A second operator gains administrative access** to either lince
   or the rsync.net account. Two-operator surface raises insider-risk
   posture; defence-in-depth becomes proportionate.
4. **A formal compliance / DR requirement is published** (internal
   policy, customer audit, regulatory) that mandates multi-target
   backup.

Until any of those triggers fire, the single-target Phase-1 model
stands. The trigger list is exhaustive for this ADR — informal
arguments ("we should probably add B2 anyway") do not move the
decision.

## Rationale

### Why rsync.net standard, not Borg-only

The Borg-only discount is a real cost reduction, but the cost being
optimized — storage at < 10 GB — is already trivial. The savings would
amount to a few dollars per year against giving up provider-side
snapshots and full SSH access. That trade is asymmetric in the wrong
direction: we'd save dollars and lose a layer of defence.

A backup that exists only in a Borg repository, with no out-of-band
snapshot behind it, is a single logical artifact. Compromise the
repository (passphrase leak, insider, ransomware that propagates from
lince to the backup host through the SSH key) and there is nothing to
fall back on. ZFS snapshots are independent of the Borg layer's
integrity.

### Why not Backblaze B2 as primary

B2 is cheaper and S3-compatible. It is a strong candidate as a 3-2-1
secondary, but rejected as Phase-1 primary on three grounds:

- Object Lock is the immutability primitive. Configuring Object Lock
  correctly (governance vs compliance retention modes, retention
  windows, IAM separation between writer and prune-actor) is high
  ceremony and rarely tested before disaster strikes. ZFS snapshots
  are a default, not a configuration.
- Backblaze is US-incorporated. The EU region addresses data residency
  but not jurisdictional reach (CLOUD Act). For a Swiss-positioned
  brand carrying institutional mail, this is a coherence cost.
- Egress charges become material on full restore. rsync.net has no
  egress fee. For an emergency restore, we want to know the cost
  upfront.

### Why not Hetzner Storage Box

Cheapest absolute cost at scale (€3.81/TB-month). The blocker is
shared-region failure correlation: Contabo and Hetzner are both German
hosting, with overlapping regional infrastructure exposure. A regional
event affecting one is plausibly correlated with the other. rsync.net
Zurich is jurisdictionally and physically distinct from Contabo's
European footprint, providing genuine off-site failure-domain
separation.

### Why not Storj

Decentralized object storage is cheap and architecturally interesting,
but introduces a less battle-tested operational story for backup
restore at scale. Backup is the wrong place to optimize for novelty.
The token-economy procurement story also complicates audit in a
regulated context.

### Why not a self-hosted secondary VPS

Operationally cheap on paper, expensive in attention. The team is
small. Adding a second VPS turns "backup target" into "host I have to
patch, monitor, and harden", which is exactly the kind of operational
surface RBX is trying to keep narrow. SaaS-managed targets win on
attention budget even if they cost slightly more in dollars.

## Consequences

### Positive

- Provider-side ZFS snapshots reduce the risk of ransomware and
  operator-error data loss. They are independent of the backup engine
  used by the client and read-only from the client's perspective, so
  a compromised lince — or a misuse of the Borg/Restic credentials
  from lince — cannot mutate the snapshot history.
- Swiss jurisdictional alignment matches rbxsystems.ch positioning.
- Toolchain stays unified: SSH, Ansible, Borg/Restic — already
  understood by the team.
- No egress fee removes a hidden cost from disaster restore.
- Single-provider Phase-1 keeps operational surface minimal.

### Negative

- Single off-site target only — does not satisfy strict 3-2-1.
  Mitigated by the deferred-secondary trigger list, but acknowledged.
- Premium pricing per-GB compared to Hetzner / B2 / Storj. At Phase-1
  volume the absolute number is trivial; this becomes worth revisiting
  if mail volume crosses ~50 GB sustained.
- Provider-side ZFS snapshot retention default is 7 days. Beyond
  7 days, only the client-side Borg/Restic policy protects. Acceptable
  given the layered model.

### When to revisit

Revisit this ADR (not necessarily change it — revisit) when any of
these conditions hold:

- A 3-2-1 trigger fires (see "Secondary 3-2-1 target" above).
- Mail volume crosses 50 GB sustained.
- A restore drill fails or takes longer than the runbook RTO.
- rsync.net materially changes its account types, pricing, or service
  posture.
- An ADR-level decision elsewhere in RBX changes the brand /
  jurisdictional posture for institutional mail.
